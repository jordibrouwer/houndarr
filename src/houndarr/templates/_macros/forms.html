{#- Shared form macros. The Houndarr web surface repeats a handful of input shapes across login, setup, admin > security, the admin confirm dialog, and the add / edit instance form. E.12 can migrate consumers batch by batch without forking the class strings or data-* markers that auth.js (caps badge, strength meter, confirm-match, show-hide toggle) and CSS (.field, .input-wrap, .input-trailing, .station-input, .has-trailing, .mono, .pw-match, .strength*) rely on. Variants are explicit. password_input branches on variant="auth" (login / setup, with the .field__label + leading SVG treatment from auth-fields.css) versus variant="admin" (security + confirm dialog, with the Tailwind utility label and the h-10 px-3 input class). The two families render through different CSS language; collapsing them into one variant would lose either the auth-card visual or the admin-page visual. Track E.6 introduces this file without migrating consumers; later batches do that. Centralising the markup here means Track G can change a form element's component class in one seam instead of seven inline call sites. Whitespace layout per macro is deliberate and pinned by tests/test_templates/test_macros_forms.py: form_field emits every attribute on a single line; password_input variant="auth" keeps the pre-refactor multi-line auth-card attribute layout; variant= "admin" keeps the one-attribute-per-line admin layout.
